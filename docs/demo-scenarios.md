# Demo Scenarios

## 范围冻结

本 MVP 明确不做以下能力：

- OEM 或系统签名级权限
- 支付、转账、下单
- 大范围多 App 复杂 UI 自动化
- 自主安装 App、修改关键系统设置
- 完整端侧大模型部署优化

本期只追求“高可演示、可闭环、可审计”的三方 App 路线。

## 场景 1：通知理解与动作执行

### 用户故事

用户收到票务或会议类通知，系统自动提取关键信息，建议后续动作，并在用户确认后执行。

### 输入

- Android 通知事件
- 设备时间
- 当前会话上下文

### 目标动作

- 读取通知内容
- 提取时间、地点、联系人、任务类型
- 生成建议动作
- 选择执行 `calendar.add`、`notifications.action` 或 `app.launch`

### 成功判定

- 可以从通知构造结构化任务
- 执行结果被写入任务日志
- 对中风险动作执行确认流

## 场景 2：日程助理

### 用户故事

用户用自然语言要求“帮我明天上午安排和张三开会并提前提醒”，系统完成事件创建与提醒策略生成。

### 输入

- 用户自然语言请求
- 已有日历事件
- 时间解析结果

### 目标动作

- 读取日历冲突
- 创建日程
- 生成出发/提醒建议

### 成功判定

- 日历事件成功写入
- 返回结构化确认信息
- 冲突或缺参时要求补充

## 场景 3：通讯助理

### 用户故事

用户说“给李雷发短信，告诉他我会晚到十分钟”，系统检索联系人、生成短信并在确认后发送。

### 输入

- 用户消息
- 联系人搜索结果

### 目标动作

- `contacts.search`
- 选择目标联系人
- 生成短信草稿
- `sms.send`

### 成功判定

- 找到联系人或给出明确歧义提示
- 中风险发送动作必须确认
- 发送回执进入任务日志

## 场景 4：UI 自动化 MVP

### 用户故事

用户要求“打开某个 App 并完成一个固定的轻量流程”，系统通过可见界面观察与动作执行完成闭环。

### 输入

- 用户目标
- `ui.inspect` 返回的界面快照
- 可选截图/OCR 文本

### 目标动作

- `app.launch`
- `ui.inspect`
- `ui.act`
- 必要时循环观察和重规划

### 限制

- 仅支持单 App、单路径、单设备分辨率验证
- 本期不追求泛化 UI Agent

### 成功判定

- 至少打通一个固定流程
- 每一步都有观测记录
- 遇到不确定状态时优先停下并请求确认
